UST Library Search × Claude

A Model Context Protocol (MCP) server that lets an AI assistant search the University of St. Thomas library catalog directly — so answers come from the library's curated records, with links back to the real items.

What this is

When students ask an AI assistant for sources, it typically answers from the open web — or invents citations. Yale and Northeastern have both piloted a fix: a small MCP server that gives the assistant a search-the-library tool. The assistant decides when to use it, the library's own discovery index does the searching, and every result carries a permalink into the catalog. Yale's student testing found this visible provenance measurably increased trust in AI outputs.

This site is that idea, working, for UST. It wraps the same public, unauthenticated search endpoint that powers librarysearch.stthomas.edu (Alma + Primo VE via CLIC/MnPALS). Nothing is scraped, no credentials are used, and no student data is stored.

Try the data it returns

This demo box calls the same backend function Claude calls. Default scope is UST's own catalog records only (see decision #1 below).

Try it inside Claude

  1. In Claude (web or desktop), open Settings → Connectors → Add custom connector.
  2. Name it UST Library Search and paste this URL: { this page's URL }/mcp
  3. Start a chat and ask: “What does the UST library have on Dante's Divine Comedy?”

Claude will call the server's two tools — search_catalog and get_record — and answer from real catalog records, citing permalinks. No login is required because the underlying catalog search is already public.

Key decisions before institutional adoption

This prototype works, but it deliberately leaves the real decisions to the library. Per Dan Cohen's framing at Northeastern: this technology should be library-driven and library-controlled.

1 · Data scope: UST records only, or the full blended index?
Primo searches can include Ex Libris's proprietary Central Discovery Index (articles and content beyond UST's own records). Whether UST may expose that index through its own AI tooling is a contract question currently with library leadership. This prototype defaults to UST's own catalog records only, which libraries are generally free to use — the blended scope exists in the code but is opt-in and clearly labeled.
2 · Which API endpoint?
The prototype uses Primo's public UI endpoint (pub/pnxs): rich results, no key, but technically internal — Ex Libris could change it. A production version should weigh the officially sanctioned external/pnxs endpoint (capped at 10 results per call) or a keyed Primo Search API arranged through CLIC/Ex Libris.
3 · Governance and hosting
Who owns, reviews, and operates the service: the library, ITS, or both? This instance runs on Google Cloud Run for about the cost of a coffee per month, but an adopted version belongs on institutionally managed infrastructure with a security review — the code is ~300 lines of Python and was written to be auditable in one sitting.
4 · Access control and privacy
The catalog data is public, so this endpoint is open. An institutional version should decide: open to the world, UST accounts only, or bundled into UST's planned AI platform? And set an explicit logging policy — search queries can be sensitive, and this prototype intentionally stores none.
5 · Which AI platforms?
MCP is an open standard: the same server works with Claude, ChatGPT, open-source models, and whatever UST's AI platform adopts. Piloting in Claude first mirrors Yale and Northeastern, but nothing here is vendor-locked — and library content accessed this way is not used for model training.
